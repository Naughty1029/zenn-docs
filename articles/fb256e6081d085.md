## はじめに

「抽象クラスとインターフェース、どっちを使えばいいの？」

オブジェクト指向を学んでいると、ほぼ確実にこの疑問にぶつかります。私自身、最初に抽象クラスに出会ったとき「中身のないクラスを作って何がうれしいの？」と思っていました。小さなコードでは確かに不要に見えるのですが、チーム開発を経験するうちに「ああ、こういうことか」と分かるようになりました。

この記事では、まず具象クラスと抽象クラスの基本を押さえたうえで、インターフェースとの違いを「設計の視点」から整理していきます。サンプルコードはPHPで書いていますが、JavaやTypeScriptなど他の言語でも考え方は共通です。

## 具象クラスのおさらい

普段書いているクラスは、基本的にすべて具象クラスです。メソッドに処理が書かれていて、そのままインスタンス化して使えます。

```php
class Dog
{
    public function cry(): string
    {
        return 'ワン！';
    }
}

$dog = new Dog();
echo $dog->cry(); // ワン！
```

処理の中身がすべて揃っているので、`new`してすぐに動きます。普段書いているクラスはほとんどこれです。

では、なぜわざわざ「具象クラス」という言い方をするのでしょうか。それは、次に説明する「抽象クラス」と区別するためです。

## 抽象クラスとは

抽象クラスは、一部のメソッドの「呼び方」だけを定義し、具体的な処理は子クラスに任せるクラスです。

```php
abstract class Animal
{
    // 具象メソッド：共通の処理をここに書ける
    public function eat(): string
    {
        return 'もぐもぐ';
    }

    // 抽象メソッド：中身は子クラスが決める
    abstract public function cry(): string;
}
```

ここで重要なのは、**抽象クラスは具象メソッドと抽象メソッドの両方を持てる**という点です。`eat()`のように共通の処理を親クラスにまとめつつ、`cry()`のように子クラスごとに異なる処理は抽象メソッドとして宣言する、という使い方ができます。

抽象クラスはそのままではインスタンス化できません。

```php
$animal = new Animal(); // Fatal error: Cannot instantiate abstract class
```

子クラスで抽象メソッドをすべて実装（オーバーライド）して初めて使えるようになります。

```php
class Dog extends Animal
{
    public function cry(): string
    {
        return 'ワン！';
    }
}

class Cat extends Animal
{
    public function cry(): string
    {
        return 'ニャー！';
    }
}

$dog = new Dog();
echo $dog->cry();  // ワン！
echo $dog->eat();  // もぐもぐ（親クラスの共通処理）
```

ここまで読むと、「具象クラスだけで同じことが書けるのでは？」と思うかもしれません。確かに小さなコードではそうです。では、なぜわざわざ抽象クラスを使うのでしょうか。

## なぜ抽象クラスを使うのか

数十人で開発する大規模なプロジェクトを想像してみてください。他の人が書いた基底クラスを継承してサブクラスを作るとき、こんな問題が起こりがちです。

- オーバーライドすべきメソッドの**実装漏れ**
- メソッド名の**スペルミス**（`cry`のつもりが`cly`になっていた、など）
- どのメソッドをオーバーライドすべきか分からず**ソースを読む時間が膨大**

抽象クラスを使えば、これらの問題をコンパイル（実行）時に検出できます。

```php
class Bird extends Animal
{
    // cry()を実装し忘れると...
    // Fatal error: Class Bird contains 1 abstract method
    // and must therefore be declared abstract
    // or implement the remaining methods (Animal::cry)
}
```

要するに、抽象クラスを使うと「このメソッドを実装し忘れてるよ」とエラーで教えてもらえるわけです。小さなコードだと実感しにくいですが、チーム開発では実装漏れやメソッド名の統一に効いてきます。

では、抽象クラスと似た仕組みである「インターフェース」はどう違うのでしょうか。

## インターフェースとは

インターフェースも、抽象クラスと同じく「メソッドの実装を強制する」仕組みです。ただし、原則として**処理の中身は持たず、メソッドのシグネチャだけを定義**します。

```php
interface Cryable
{
    public function cry(): string;
}
```

使う側は`implements`キーワードで宣言します。

```php
class Dog implements Cryable
{
    public function cry(): string
    {
        return 'ワン！';
    }
}
```

ここまでだと抽象クラスとほぼ同じに見えます。違いを具体的に見ていきましょう。

## 抽象クラスとインターフェースの違い

### インターフェースは複数実装できる

PHPやJavaではクラスの継承は1つだけですが、インターフェースは複数同時に実装できます。

```php
interface Flyable
{
    public function fly(): string;
}

interface Swimmable
{
    public function swim(): string;
}

class Duck implements Flyable, Swimmable
{
    public function fly(): string
    {
        return 'バサバサ';
    }

    public function swim(): string
    {
        return 'スイスイ';
    }
}
```

抽象クラスではこうはいきません。

```php
// これはエラー（PHPは単一継承のみ）
class Duck extends Bird, WaterAnimal { }
```

「空を飛べる」「泳げる」のような能力を組み合わせたいとき、インターフェースが使えます。

### 抽象クラスは共通処理を持てる

先ほどの`Animal`クラスでも見たように、抽象クラスはプロパティや具象メソッドを持てます。子クラス間で共通の処理をまとめておけるのが強みです。

```php
abstract class BaseController
{
    protected string $layout = 'default';

    // 共通処理
    public function renderLayout(): string
    {
        return "レイアウト: {$this->layout}";
    }

    // 子クラスごとに異なる処理
    abstract public function handle(): string;
}

class UserController extends BaseController
{
    public function handle(): string
    {
        return 'ユーザー一覧を表示';
    }
}
```

インターフェースは原則として処理を持たないため、こうした共通処理の共有には向いていません。

これらの違いを表にまとめると、次のようになります。

|  | 抽象クラス | インターフェース |
|---|---|---|
| 役割 | 共通の処理＋ルール | ルール（契約）だけ |
| メソッドの実装 | 具象・抽象の両方OK | 原則なし |
| 継承・実装 | 1つだけ（単一継承） | 複数OK |
| プロパティ | 持てる | 定数のみ |
| 使いどころ | 共通の土台を作りたいとき | 能力・役割を定義したいとき |

## 使い分けの指針

では実務ではどう使い分けるのか、ざっくりした目安です。

**インターフェースを使うとき**は、「何ができるか（能力・契約）」を定義したい場面です。

- `Payable`（支払いできる）
- `Loggable`（ログ出力できる）
- `JsonSerializable`（JSON化できる）

異なる種類のクラスに共通のルールを与えたいときに適しています。

**抽象クラスを使うとき**は、「共通のベースを作りたい」場面です。

- `BaseController`（コントローラーの共通処理）
- `AbstractRepository`（データ取得の共通ロジック）

子クラス間で処理を共有しつつ、一部だけ個別実装させたいときに向いています。

:::message
迷ったらまずインターフェースを検討してみてください。インターフェースのほうが制約が少なく、後から設計を変更しやすいためです。共通処理の共有が本当に必要だと分かった段階で抽象クラスを導入しても遅くはありません。
:::

## まとめ

- **具象クラス**はそのままインスタンス化して使える、普段書いているクラス
- **抽象クラス**は共通処理を持ちつつ、子クラスに「このメソッドは必ず実装しろ」と強制できる
- **インターフェース**は処理を持たず、メソッドのシグネチャだけを定義する。複数同時に実装できる
- 能力・役割の定義にはインターフェース、共通処理の共有には抽象クラス
- 迷ったらインターフェースから始めてみる

小さなコードだと「回りくどいな」と感じると思いますが、チーム開発で規模が大きくなると実装漏れの防止やコードの統一に効いてきます。まずは手元で動かしてみてください。
